§00 Hospital Operating System

Your hospital doesn't need more software. It needs an operating system.

Every hospital management system on the market is a filing cabinet with a login screen — forty-five modules, four hundred screens, and a database that only remembers the last thing that happened. Ospia HOS is built the other way round: intent instead of navigation, a permanent event history instead of last-known-state, configuration instead of code, and AI that is required to explain itself before anyone acts on it.

§01 Why not another HMS

An HMS records what happened. An operating system runs the place.

We built one of the good ones first — and then went looking for the ceiling. We found three, and none of them could be patched out.

Ceiling one

The screen is the product

When capability lives in pages, every new channel — mobile, kiosk, voice, an agent, a partner's API — is a rebuild. The work is real but it is always the same work, done again in a new shape.

Ceiling two

The database forgets

A row holds one truth: the current one. Ask what a chart looked like at the moment of the decision, or replay how a claim reached denial, and the honest answer is that nobody kept it.

Ceiling three

Every hospital is a fork

Numbering schemes, bed policies, discount matrices, approval chains — each hospital's rules end up in someone's code. Ten customers become ten codebases and a support burden that compounds.

Ospia HMS — the system that taught us this — is live, sold, and supported. HOS is not its next version; it is its successor architecture, rebuilt greenfield so that the ceilings are absent rather than worked around. What v1 proved about clinical breadth, safety and Indian compliance is inherited wholesale. What it proved about platform mechanics is the reason for the rewrite.

§02 The platform map

Ten platforms, not forty-five modules

Modules are how work gets organised. Platforms are what the work compounds into. Everything Ospia ships lands in one of these ten, and each one is reusable by every other.

Identity, authorization, tenancy, configuration, crypto, observability — the kernel, deliberately tiny, with zero business rules.

02

Patient, EMR, orders, nursing, ICU, theatre, beds, emergency, lab, imaging, blood bank — one record, one timeline.

03

Billing and RCM, insurance and claims, and a real general ledger — statutory accounting, not an export.

04

AI

Agent runtime, memory, model router, decision ledger and governance — the intelligence layer, governed like staff.

05

Integration

FHIR, HL7, ABDM, DICOM, LIS/RIS, ERP, payments, messaging, devices — a connector catalog, configured not coded.

06

Analytics

Read-models built from events, a live digital twin, a prediction service, and an export path to a data lake.

07

Automation

Workflow engine with timers, SLAs, escalation and compensation — plus the rules engine everything routes through.

08

Developer

Typed contracts, generated OpenAPI and SDK, a published event catalog, documented extension points.

09

Marketplace

Every module is an installable package with a manifest. A store is a storefront over a mechanism that already exists.

10

Experience

A design system, role workspaces, the command bar, the operations wall — thin surfaces over a headless core.

The rule that keeps this honest: platform before features. Before anything is built we answer eight questions — can another feature reuse it, can AI use it, can automation use it, can workflows orchestrate it, can external systems integrate with it, can hospitals configure it, does it add to institutional knowledge, will it still make sense in ten years. Any "no" sends the design back. It is slower on any given Tuesday and it is the only reason the tenth capability costs less than the first.

§03 Intent & the task registry

Tasks are permanent. Screens are temporary.

The most consequential decision in the whole architecture, and the least visible one: capability does not live in pages.

The registry

Every capability is a declared task

Register Patient. Admit Patient. Prepare Theatre. Dispense Medicine. Generate Claim. Book Follow-up. Each is registered by its module with typed parameters, the permission it demands, the workflow that orchestrates it, and the events it publishes.

typed params · required permission · orchestration · emitted events · autonomy level
The consequence

Six front doors, one implementation

A screen, the ⌘K bar, a voice instruction, the AI copilot, an autonomous agent and a partner's API call are all invokers of the same task. Not six code paths that must be kept in agreement — one, with six ways in.

screen · command bar · voice · copilot · agent · external API
The payoff

Every screen eventually optional

"Generate today's discharge bills" resolves to a task invocation, not a navigation path. New surfaces — kiosk, ward tablet, smart-TV wall, wearable, whatever 2032 brings — are thin clients over a registry that already exists.

intent is the interface · screens are one representation of it

This is also what makes the AI safe to expand. An agent is not given a browser and a login — it is given a permission-scoped subset of the same task registry your staff use, under the same authorization checks, producing the same audited events. There is no back door because there is no second door.

§04 The autonomy ladder

Automation is a dial, and the hospital holds it

Every task is designed to be capable of full autonomy, and every task ships set to whatever level you are actually comfortable with. Moving the dial is a configuration change — reviewed, approved and audited like any other.

Manual

The dial is independent of risk. Risk classifies the action; maturity configures how far automation may currently go for that action, at your hospital. A low-risk task may sit at L4 on the day you go live; a high-risk one may sit at L1 for two years and then move — because you moved it, on evidence, with a signature attached. Nothing promotes itself.

§05 Events, time & memory

A hospital that remembers everything, and can prove it

Every action publishes an event. Every event is kept — not for ninety days, not until the next archive job. Permanently. The event log is the hospital's history.

Events over coupling

One fact, many listeners

Pharmacy publishes MedicineDispensed and knows nothing about who cares. Inventory decrements. Billing posts. Analytics counts. Audit chains. Adding an eleventh listener later changes nothing that already works.

durable outbox · at-least-once · idempotent consumers · versioned catalog
Think in time

As-of, not just as-is

Clinically and financially significant records are versioned append-only — no destructive updates. So "what did this chart show when the drug was prescribed" and "what did the balance sheet say on the 31st" are queries, not archaeology.

append-only version history · as-of queries · full replay
Hospital memory

Context arrives before you ask

Open a patient and the relevant past assembles itself: prior admissions, this doctor's preferences, similar cases and their outcomes, previous medications, billing and insurance history. Institutional knowledge, not a search box.

knowledge graph · memory stores · consent- and permission-gated
Reversibility

Nothing important is irreversible

Undo on user actions. Rollback and compensation on workflows. Version history and recovery on records. Replay on AI decisions. A system people trust is one where a mistake is a Tuesday, not a catastrophe.

undo · rollback · compensation · recovery · time travel
Configuration

Your rules are data, not our code

Numbering schemes, bed allocation, pricing and discounts, package rules, approval matrices, referral and insurance rules, consent requirements, clinical pathways, discharge criteria — versioned, tenant-scoped, evaluated by one engine.

target: 90% of hospital requirements met by configuration alone
Intelligent objects

Objects that know things

A patient exposes risk, timeline, predictions and care gaps. A bed exposes occupancy, cleaning, maintenance and the equipment near it. A medicine exposes alternatives, interactions, demand and expiry. Behaviour and insight, not just columns.

insight surface on every contract · consumed by cards, twin, agents alike

§06 The trust layer

AI that has to show its working

Not a policy commitment — a response contract. Nothing reaches a clinician or an administrator without answering five questions, and nothing that ever happened stops being reconstructable.

AI response envelopeinventory · reorder policy

Recommendation

Raise the reorder level for Amoxicillin 500 mg from 400 to 620 units.

Why?

Consumption over 60 days is up 34%; at the current threshold cover falls to 11 days before the next scheduled delivery window.

Evidence

1,842 dispensing events · 3 supplier lead-time records (mean 6.2 days) · 2 stockouts in the last quarter.

Confidence

0.86 — lead-time variance is the dominant source of uncertainty.

Alternatives considered

Hold at 400 and shorten the delivery cycle · raise to 520 with a weekly review.

Policy references

Rules engine — inventory.reorder.service-level = 0.98, formulary class B.

Expected outcome

Stockout probability falls from 0.19 to 0.02; holding cost rises ~₹9,400 a quarter.

ledger #4a1f…9c2e · model, prompt version and inputs recorded · replayable
Decision ledger

Every AI action is replayable

Model, model version, prompt version, the exact inputs, the output, who approved it and what executed. Not a log line — a record complete enough to run the decision again and get the same answer. That is what makes an AI action auditable rather than merely logged.

Bring your own AI

No provider is named in our code

All model access goes through one router with a provider registry and a routing policy per task class — by cost, latency, language, data sensitivity and residency. Switching provider, or running a private model for sensitive classes, is configuration. Nothing in the platform knows or cares which vendor answered.

Governance

Agents are staff, and staffed accordingly

Each digital employee carries an identity, a defined scope of responsibility, permissions, tools, a schedule, KPIs, escalation rules and an audit trail. A model registry, versioned prompts, evaluation sets and performance monitoring sit behind them. Governed the way you would govern a new hire, because that is what they are.

§07 Architecture

Boundaries a code review can't be talked out of

Boundary gate

A module may import itself, the contracts package, and the kernel. Anything else fails the build.

SQL ownership gate

A repository referencing a table outside its own schema fails the build. No allow-list, no exceptions.

Layering gate

Domain logic may not import infrastructure, presentation, or any I/O library. Purity is checked, not requested.

Kernel purity gate

The kernel may not depend on any module or contract. A "utility" naming a domain noun belongs in that domain.

Contract discipline gate

Contracts carry no runtime dependencies; breaking changes to types or event schemas are detected on every pull request.

Registry completeness gate

A module ships with its tasks, permissions, events, AI tools, flags and seven documents registered — or it does not ship.

Alongside these run the ordinary ones — strict types, both-ways migrations, coverage floors, contract-versus-spec tests, an authorization matrix, accessibility scans, performance budgets, dependency and secret scanning. The ladder only ever grows. Architecture written in a document decays; architecture written in a pipeline holds.

§10 Straight answers

What committees actually ask

Should we wait for HOS, or buy HMS now?

Buy now. HMS v1 is a complete, supported hospital management system and waiting costs you a year of the benefit. Migration to HOS is a first-class engineering programme on our side, not your problem to solve: tooling with checksum verification, a parallel run where both systems process the same days and get reconciled every morning, and a per-hospital cutover with a rollback path. We do not cut anyone over on optimism.

Who owns our data?

You do, in the plainest sense. Each hospital runs on its own physical database — self-hosted if you prefer. Everything is exportable, the FHIR R4 API is open to you, and the permanent event log is yours as much as the tables are. Leaving should be as open as arriving, or the promise is not worth much.

What can the AI never do?

Act outside the task registry, or act above the autonomy level you configured. Agents invoke the same permission-checked tasks your staff do, under the same authorization, producing the same audited events — there is no privileged path. Above L2, execution requires a named human. At L4, execution requires a policy you approved, and every run still lands in the decision ledger, replayable.

There is also no silent promotion. Moving a task up the ladder is an audited configuration change with a signature attached.

What happens when the AI model is down — or you switch providers?

The hospital keeps running. Ospia is a complete system with no external AI configured at all: deterministic fallbacks cover the scribe, patient-facing explanations and the assistive paths. AI adds speed and foresight; it is never a dependency. And because no provider is named anywhere outside the model router, switching providers — or routing sensitive task classes to a private model — is a configuration change, validated by the evaluation sets.

How much of this is real versus roadmap?

Section nine answers that without hedging: v1 is live and sellable; v2 is in build with phases zero and one complete. The seams described here — the task registry, permanent event retention, persistence ports, the model router, the decision ledger, the rules engine, module manifests — are deliberately built first, because they are cheap now and impossible to retrofit. The surfaces they enable (marketplace, studio, data lake, ambient capture) are staged after cutover, and we say so rather than implying otherwise.

We have unusual requirements. How much gets customised?

The target is that 90% of what a hospital asks for is configuration, not code — and we measure it rather than assert it: every pilot change request gets classified config-or-code, and a code answer is treated as a design failure to be examined. Numbering, allocation policies, pricing, approval matrices, pathways, consent rules and discharge criteria are versioned data evaluated by one engine, not conditionals in a fork of your own.

Can we integrate our LIS, PACS and ERP?

Yes, through a connector catalog rather than a project: FHIR R4 and HL7v2, DICOM/PACS, LIS/RIS, ERP and accounting, HRMS, government portals, payment gateways, WhatsApp/SMS/email, and device feeds. Each connector declares its own configuration schema, so onboarding one is a configuration exercise. Outbound, every domain event can be subscribed to as a signed webhook.
